# FIDES Grow Alternative Credit Schema

## Decision

Grow should include a mocked alternative-credit profile that combines trust graph evidence and vnSocial reputation.

This is a Grow schema update because the profile is not raw invoice input. It is derived intelligence about the business across many relationships: repeat buyers, verified counterparties, cashflow stability, and public/social reputation.

For the 5-day MVP, this is deterministic mock data. It lets the demo show how a thin-file merchant can be evaluated beyond one invoice without claiming that FIDES has a production credit model or live social-data integration.

## Field Meaning

| Field | Type | Meaning |
| --- | --- | --- |
| `trust_graph_score` | number from 0 to 1 | Strength of the business relationship graph. |
| `repeat_counterparty_count` | integer | Number of counterparties with repeated transactions. |
| `verified_counterparty_count` | integer | Number of counterparties that are verified or trusted in the graph. |
| `network_centrality_score` | number from 0 to 1 | Whether the business sits in a healthy transaction network rather than one isolated edge. |
| `cashflow_stability_score` | number from 0 to 1 | Stability of inflow/outflow patterns from ledger and transaction evidence. |
| `vn_social_reputation_score` | number from 0 to 1 | Mock reputation score from social/public mentions. |
| `vn_social_mentions_30d` | integer | Number of recent relevant mentions. |
| `vn_social_sentiment` | string | Mock sentiment label such as `positive`, `mixed`, or `negative`. |
| `vn_social_complaint_count_30d` | integer | Recent complaint or negative-report count. |
| `alternative_credit_score` | integer from 0 to 100 | Combined mock score used as a light scoring signal. |
| `confidence` | number from 0 to 1 | Confidence in the fused profile. |
| `signals` | string array | Human-readable feature labels for explanation. |
| `explainability` | object | Gradient-boosting and SHAP-style explanation metadata. |

## Explainability Block

The explainability block gives the demo a regulator-friendly story for how the alternative score was formed.

| Field | Type | Meaning |
| --- | --- | --- |
| `model_type` | string | Model family. For MVP, fixed to `gradient_boosted_trees`. |
| `model_version` | string | Version identifier for audit and reproducibility. |
| `baseline_score` | integer from 0 to 100 | Starting point before feature contributions. |
| `final_score` | integer from 0 to 100 | Final alternative credit score explained by the features. |
| `reason_codes` | string array | Short top-factor labels for user or regulator summaries. |
| `feature_contributions` | object array | SHAP-style per-feature contribution list. |

Each `feature_contributions` item contains:

- `feature`: model feature name.
- `value`: observed feature value.
- `shap_value`: mock contribution to the score.
- `direction`: `positive`, `negative`, or `neutral`.
- `reason`: human-readable explanation of why the feature matters.

For MVP, these are SHAP-style mock values generated by deterministic formulas. They are not calculated by a real trained gradient-boosted-tree model yet.

## MVP Scoring Behavior

The Grow scorer uses this profile as a small supporting signal:

- Strong profile, score at least `80`, and no more than one recent complaint: add `10`.
- Emerging profile, score at least `65`: add `6`.
- Weak profile, score below `45`, or four or more complaints: subtract `8`.

Invoice evidence still dominates the MVP score. The alternative profile should not override a bad payment record or missing revenue evidence.

## Mock Data Rules

The synthetic generator creates category-based profiles:

- `strong_business` and `high_volume`: high trust graph, repeat counterparties, positive vnSocial reputation.
- `emerging_thin_file`: thin but plausible graph, small number of mentions, mixed sentiment.
- `late_payment`: weaker cashflow stability and a few complaints.
- `seasonal_cashflow`: medium graph strength with seasonal revenue labels.

The frontend also builds a simple mock profile from the selected invoice so manual demo submissions include this block.

## Real-Life Capability

A production version would need:

- consent and clear data-use disclosure for social/reputation data,
- bank-side graph ingestion from transactions, invoices, devices, accounts, and counterparties,
- entity resolution so one business is not split across aliases, tax IDs, phone numbers, or bank accounts,
- explanation and audit logs for every score component,
- monitoring for bias, popularity effects, and sector/geography skew,
- a clean separation between positive reputation signals and unverified public accusations,
- a human-review path before adverse credit decisions.

vnSocial should be treated as a reputation and trend signal, not as a standalone credit verdict.
